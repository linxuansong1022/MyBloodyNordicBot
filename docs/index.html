把终端 AI 编码助手做“小”，但把主闭环做“全”

MiniCode 是一个面向本地开发工作流的轻量终端编码助手。它专注保留最关键的执行路径： model -> tool -> model 主循环、可审阅的文件修改流程、权限审批边界、可扩展的 Skills 与 MCP。 相比重型平台，MiniCode 更适合作为学习样本、实验底座和二次开发起点。

功能特性（基于项目文档整理）

从 README、ARCHITECTURE 与 PATTERNS 文档提炼的核心能力与设计方向。

核心工作流

  • 单轮支持多步工具执行
  • 模型与工具闭环：model -> tool -> model
  • 全屏终端交互、输入历史与 transcript 滚动
  • progress 状态与 final 回复分离建模

安全与可控

  • 文件修改前统一展示 unified diff
  • 路径权限与命令权限检查
  • 命令执行与编辑审批在主执行路径内
  • 支持拒绝并向模型反馈修正建议

扩展能力

  • 本地 SKILL.md 发现与 load_skill
  • stdio MCP server 动态接入
  • MCP tools/resources/prompts 通用访问
  • 兼容 Content-Length / newline-json framing

一轮任务如何执行

把权限与审阅放进执行链路，而不是后置补丁。

1. 接收用户请求并组织当前上下文
2. 模型决定是否调用工具并执行
3. 工具结果回传模型继续推理
4. 涉及写入时先 review diff，经审批后落盘

架构地图（当前实现）

MiniCode 不是“大而全”，而是围绕主 loop 与边界能力做结构化裁剪。

src/index.ts

CLI 入口，负责启动会话与主流程。

src/agent-loop.ts

多步模型/工具循环，驱动主要执行闭环。

src/tool.ts

工具注册、参数校验、执行入口与结果归一。

src/mcp.ts

stdio MCP 连接、协议协商、远端工具封装。

src/skills.ts

本地 skills 扫描、发现与加载。

src/permissions.ts + src/file-review.ts

路径/命令/编辑审批与写入前 diff review。

src/background-tasks.ts

后台 shell task 最小注册表，区分前台工具与后台任务。

src/tui/*

transcript、input、markdown、screen 等终端 UI 组件。

src/manage-cli.ts + src/config.ts

MCP/skills 管理命令与运行时配置加载。

通过 MiniCode 可学习的 Claude Code 风格设计

Loop-first 主结构

把 UI、工具、权限、扩展能力都围绕同一 agent loop 组织。

结构化消息模型

区分 assistant/progress/tool-call/tool-result/context-summary 等状态。

Tool Use 协议化

模型声明、系统校验、统一执行、结果标准化回流。

权限在执行路径内

危险行为通过审批与 review 进入主循环，而非旁路处理。

MCP 动态能力接入

tools/resources/prompts 在会话期动态挂载。

前台工具 vs 后台 shell task

长时命令与普通同步工具区分建模，避免伪装成“永远 running”。

路线图优先级

以下内容来自项目 ROADMAP，按 P0/P1/P2 展示。

P0（优先补齐）

  • 模型感知上下文管理与压缩
  • API retry / backoff / Retry-After
  • 会话持久化与恢复
  • Python / Go / Rust 多语言实现分支探索

P1（稳定后推进）

  • 分层 memory 加载
  • 更完整 provider abstraction
  • 轻量 Todo/task tracking
  • .claude/agents 与 sub-agent 支持
  • 有选择地扩充核心工具集

P2（中长期）

  • Notebook 编辑支持
  • 内置 WebFetch / WebSearch 工具
  • 评测与 trace 基建
  • Prompt caching

贡献方式（Contributor Friendly）

项目欢迎 PR，但强调轻量、可读、可 review 与方向一致性。

贡献原则

  • 优先小步、聚焦型 PR，避免大重构
  • 保持与 Claude Code 设计方向趋同
  • 不随意削弱安全边界（审批/权限/review）
  • 可读性优先于“技巧化”实现
  • 谨慎新增依赖，保持项目轻量

提交前检查

  • 行为改动同步更新文档
  • 先看 issue，避免重复开发
  • 本地执行 npm run check
  • PR 描述清楚改动内容、原因和验证方式
npm run check
